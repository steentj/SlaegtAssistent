# Feature: Markdown Editor & Live Preview

## Scope
Implement the right-hand pane of the shell (from feature 01) as a working Markdown editor with a
"Live Preview" tab, per `specs/Roadmap.md` Trin 1: "Implementer Markdown-editor med
'Live Preview'-faneblad" and `specs/TechStack.md` Afsnit 2 (`Markdig` for Markdown→HTML).

## In Scope
- A text editor tab bound to the currently selected person's Markdown content (plain text editing,
  load/save to the file on disk generated by feature 03).
- A "Preview" tab that renders the editor's current Markdown content as HTML via `Markdig`,
  updating as the user types (live preview).
- ViewModel logic (`EditorViewModel`) that owns the raw Markdown text and exposes a computed HTML
  string for the preview, fully unit-testable without any Avalonia rendering.
- Basic save behavior: writing edited content back to the person's `.md` file.

## Out of Scope
- Rich-text/WYSIWYG editing (Trin 1 explicitly calls for plain Markdown text + a separate preview
  tab, not an inline rendered editor).
- AI enrichment button/integration (Trin 3, out of this sprint).
- Auto-save/debounced disk writes beyond an explicit "Save" action (can be revisited later if
  needed — Trin 1 only requires the editor and preview to function).
- Syntax highlighting in the raw Markdown editor (nice-to-have, not required by the roadmap).

## Decisions & Context
- **HTML rendering library:** `Markdig` (per `TechStack.md`), used purely to convert the current
  Markdown string to an HTML string. The `EditorViewModel` owns this conversion in a computed
  property so it can be unit-tested by asserting on the resulting HTML string, without needing to
  actually render it in a UI control.
- **Preview display:** The HTML string is displayed via an Avalonia HTML view control (per
  `TechStack.md`'s mention of an `AvaloniaWebView`-style element). The exact control choice is a
  View-layer detail; the ViewModel only needs to expose the HTML string, keeping the feature
  testable independent of which control ends up rendering it.
- **Data flow:** `EditorViewModel.MarkdownText` (settable, e.g. bound to a `TextBox`) →
  `EditorViewModel.PreviewHtml` (computed via `Markdig.Markdown.ToHtml(MarkdownText)`, recomputed
  whenever `MarkdownText` changes).
- **File I/O:** `EditorViewModel` is constructed with a file path (from the selected person) and
  exposes `Load()`/`Save()` methods. File I/O is isolated behind a small `IMarkdownFileStore`
  abstraction so `EditorViewModel` tests can use an in-memory fake instead of touching disk.
- **Why isolated:** This feature can be developed and unit-tested entirely with hand-crafted
  Markdown strings and a fake file store — it does not need a real GEDCOM file, a real generated
  biography, or the actual two-pane shell to exist first, though it does plug into the `TabControl`
  placeholder created in feature 01.

## Dependencies
- Requires the `TabControl` placeholder from feature 01 (project scaffold) to attach the Editor
  and Preview tab content to. Can consume `.md` files produced by feature 03, but its own tests use
  a fake file store, so it can be developed in parallel with features 02/03.
